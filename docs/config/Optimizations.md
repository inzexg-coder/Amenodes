# `Optimizations`

## ОПИСАНИЕ

`Optimizations` – конфигурационный модуль, определяющий доступные оптимизации производительности для приложения Amenodes.
Каждая оптимизация представляет собой настраиваемый компонент, который может быть включён/выключен пользователем через панель оптимизации или использован системой бенчмаркинга для измерения прироста производительности.

Модуль экспортирует единственную константу `OPTIMIZATIONS` – массив объектов, каждый из которых описывает конкретную технику оптимизации рендеринга, вычислений или управления памятью.

**Важно:** данный модуль не содержит логики применения оптимизаций – он только предоставляет метаданные.
Реализация каждой оптимизации находится в соответствующих классах: `DomRenderer`, `BenchmarkService`, `History`, `Viewport` и других.

# ЭКСПОРТЫ

## OPTIMIZATIONS

```javascript
export const OPTIMIZATIONS: Array<OptimizationDefinition>
```

Массив определений оптимизаций, доступных в системе. Каждый элемент имеет следующую структуру:

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `name` | `string` | Да | Отображаемое имя оптимизации (на русском языке). |
| `desc` | `string` | Да | Краткое описание технической реализации. |
| `pros` | `string` | Да | Описание преимуществ – что ускоряется или улучшается. |
| `cons` | `string` | Да | Описание недостатков – какие проблемы или ограничения возникают. |
| `impl` | `boolean` | Да | Флаг реализации: `true` – оптимизация полностью реализована, `false` – заглушка/планируется. |
| `type` | `string` | Нет | Тип элемента управления. Отсутствие или `undefined` означает чекбокс; `"slider"` – ползунок для числового параметра. |
| `min` | `number` | Для `type="slider"` | Минимальное значение ползунка. |
| `max` | `number` | Для `type="slider"` | Максимальное значение ползунка. |
| `step` | `number` | Для `type="slider"` | Шаг изменения значения ползунка. |
| `default` | `number` | Для `type="slider"` | Значение по умолчанию (в процентах). |

# СПИСОК ОПТИМИЗАЦИЙ

## Виртуализация (Virtualization)

**Индекс в массиве:** `0`

**Техническая реализация:** метод `setVirtual()` в `DomRenderer`.
При включении рендерер отрисовывает только узлы, попадающие в видимую область viewport с отступом `margin = 300` пикселей. Остальные узлы удаляются из DOM, но сохраняются в `elementCache` для быстрого восстановления при скролле.

**Преимущества:**
- Уменьшение количества DOM-элементов на странице вплоть до 90% при большом количестве узлов.
- Ускорение операций reflow и repaint.
- Снижение потребления оперативной памяти.

**Недостатки:**
- При быстром скролле может наблюдаться кратковременное мерцание при подгрузке узлов.
- Необходимость вычислять видимость каждого узла при каждом рендере (дополнительная вычислительная нагрузка).
- Кэшированные элементы в `elementCache` продолжают занимать память.

## GPU-трансформации (GPU Transforms)

**Техническая реализация:** применение CSS-трансформации `translate3d(0,0,0)` к контейнеру `.canvas-container`. 
Эта техника форсирует создание отдельного композитного слоя для контейнера, что позволяет GPU обрабатывать трансформации независимо от основного потока рендеринга.

**Преимущества:**
- Анимации панорамирования и масштабирования обрабатываются на GPU, достигая 60 FPS даже при низкой производительности CPU.
- Освобождает основной поток браузера для выполнения JavaScript-логики.
- Особенно эффективен на мобильных устройствах с ограниченным CPU.

**Недостатки:**
- Увеличение использования видеопамяти (VRAM) из-за создания дополнительных композитных слоёв.
- На некоторых GPU (особенно старых или интегрированных) могут возникать графические артефакты: мерцание, разрывы линий, некорректное наложение.
- Может конфликтовать с другими CSS-свойствами, влияющими на композитинг.

## Кэш высоты (Height Cache)

**Техническая реализация:** хранение вычисленных высот узлов в `Map` (`this.heightCache` в `DomRenderer`). При каждом рендере вместо вызова `node.getMinHeight()` используется кэшированное значение.
Инвалидация кэша происходит через `invalidateCache(nodeId)` при изменении содержимого узла.

**Преимущества:**
- Ускорение вычислений позиционирования узлов, особенно при частых рендерах.
- Снижение количества вызовов методов узлов, которые могут выполнять сложные вычисления.
- Особенно эффективно для узлов с динамической высотой.

**Недостатки:**
- Необходимость ручной инвалидации кэша при изменении содержимого узла – если забыть вызвать `invalidateCache`, визуальные размеры узла могут устареть.
- Дополнительное потребление памяти для хранения кэша.
- Начальное заполнение кэша требует первого вычисления.

## will-change

**Техническая реализация:** установка CSS-свойства `will-change: left, top` на DOM-элементы узлов через метод `applyOptStyles` в `DomRenderer`. 
Это свойство предупреждает браузер о том, что свойства `left` и `top` будут изменяться, позволяя браузеру заранее оптимизировать эти элементы.

**Преимущества:**
- Браузер создаёт отдельные композитные слои для узлов, что ускоряет анимацию перемещения.
- Снижение нагрузки при перетаскивании узлов.
- Улучшение плавности при изменении позиций.

**Недостатки:**
- Каждый элемент с `will-change` потребляет дополнительную память (отдельный слой).
- При использовании на большом количестве узлов (1000+) потребление памяти может стать проблемой.
- Чрезмерное использование может привести к обратному эффекту – замедлению из-за создания слишком большого количества слоёв.

## CSS containment

**Техническая реализация:** установка CSS-свойства `contain: layout paint` на DOM-элементы узлов через метод `applyOptStyles`. 
Это свойство изолирует узел от остальной страницы, сообщая браузеру, что внутренние изменения узла не влияют на макет соседних элементов.

**Преимущества:**
- Браузер может пропустить полный перерасчёт макета при изменении узла.
- Ускорение операций reflow при изменении содержимого узлов.
- Особенно эффективно для узлов с динамическим содержимым (`GroupNode`, `OutputNode`).

**Недостатки:**
- Свойство `contain: layout` может обрезать содержимое, выходящее за пределы узла.
- Несовместимо с некоторыми CSS-свойствами.
- Может вызвать проблемы с позиционированием выпадающих элементов.

## Pointer-events линий (Pointer-events Lines)

**Техническая реализация:** настройка области клика по линиям через CSS-свойство `pointer-events: visibleStroke`.
Это позволяет срабатывать событиям мыши только на видимом контуре линии, а не на всей её ограничивающей рамке. В `EdgeRenderer` у линий установлено `style.pointerEvents = "visibleStroke"`.

**Преимущества:**
- Ускорение обработки кликов на линиях – браузер не проверяет столкновения с прозрачной областью.
- Уменьшение количества вычислений hit-test при кликах.
- Предотвращение ложных срабатываний при клике рядом с линией.

**Недостатки:**
- Прирост производительности незначителен для большинства сценариев.
- Может затруднить выделение линий с очень тонкой обводкой.
- Тонкие линии (толщиной 1-2 пикселя) становится сложнее попасть курсором.

## Ограничение истории (Limit History)

**Техническая реализация:** изменение максимального размера стека истории в классе `History` с 50 шагов до 20 (`history.maxSize = 20`).

**Преимущества:**
- Снижение потребления оперативной памяти на 60% для стека истории.
- Уменьшение размера автосохранения в localStorage.
- Более быстрое выполнение `undo` и `redo` из-за меньшего стека.

**Недостатки:**
- Пользователь может отменить только 20 последних действий вместо 50.
- При интенсивной работе история может быть перезаписана быстрее.
- Невозможно вернуться к состоянию, которое было более 20 действий назад.

## Качество дизайна (Design Quality)

**Техническая реализация:** управление CSS-классами на элементе `document.body` в зависимости от значения ползунка. Метод `applyDesignQuality` в `Application` устанавливает следующие классы:

| Диапазон | Класс | Визуальные изменения |
|----------|-------|-----------------------|
| `≤ 20%` | `design-quality-extreme` | Полное отключение теней, скруглений, градиентов, backdrop-filter, переходов. |
| `21-50%` | `design-quality-1` | Упрощённые тени, минимальные скругления, отключение размытия. |
| `51-80%` | `design-quality-2` | Умеренное упрощение, сохранение базовой эстетики. |
| `81-100%` | (нет класса) | Полный визуальный режим со всеми эффектами. |

**Преимущества:**
- Экстремальный прирост FPS – до +300% на маломощных устройствах.
- Адаптация под производительность устройства – пользователь может настроить баланс.
- Особенно эффективно для графов с 100+ узлами на встроенных видеокартах.

**Недостатки:**
- Значительное ухудшение визуального качества интерфейса.
- При экстремальном режиме интерфейс выглядит как примитивный wireframe.
- Стилизация становится непредсказуемой для пользователя.
- 
### Отображение прироста FPS для качества дизайна

При изменении слайдера качества дизайна отображается предполагаемый прирост FPS:

| Качество | Диапазон | FPS Gain |
|----------|----------|----------|
| Extreme | ≤ 20% | +300% |
| Low | 21-50% | +150% |
| Medium | 51-80% | +50% |
| High | 81-100% | 0% (реальный прирост после бенчмаркинга) |

При применении качества дизайна автоматически запускается пересчёт бенчмаркинга для отображения актуального FPS.


# РАСШИРЕНИЯ

Для добавления новой оптимизации необходимо:

1. Добавить объект в массив `OPTIMIZATIONS`:
```javascript
{
  name: "Новая оптимизация",
  desc: "Техническое описание",
  pros: "Преимущества",
  cons: "Недостатки",
  impl: true  // или false, если не реализована
}
```

2. Реализовать логику включения/выключения в `DomRenderer`, `BenchmarkService` и других модулях.

3. Для оптимизации-ползунка добавить `type: "slider"` и параметры `min`, `max`, `step`, `default`.

4. Для отображения в панели оптимизации добавить соответствующие ключи переводов в `en.js` и `ru.js` (секция `optimizations`).

## ЗАМЕЧАНИЯ

- Индексы оптимизаций в массиве фиксированы и используются для идентификации в `BenchmarkService.applyOptimizationByIndex`. Изменение порядка требует обновления этого метода.
- Оптимизации с `impl: false` отображаются в панели, но отключены (стиль `opt-item-disabled`). Это позволяет пользователю видеть планируемые функции.
- Оптимизация "Качество дизайна" обрабатывается отдельно через `applyDesignQuality` и не участвует в автоматическом бенчмаркинге (пропускается через `if (opt.type === 'slider') continue`).
- Реальные значения прироста FPS для каждой оптимизации вычисляются динамически `BenchmarkService` и хранятся в `this.realGains`.
